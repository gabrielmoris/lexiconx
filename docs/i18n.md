# Internationalization (i18n)

## Locale ↔ Language Mapping

The app uses two related but distinct type systems for language:

| Locale (URL/routing) | Language (data/AI) |
|----------------------|---------------------|
| `en` | `English` |
| `de` | `Deutsch` |
| `zh` | `中文` |
| `es` | `Español` |
| `ru` | `русский` |

- **`Locale`** — used in URLs, file names, next-intl routing (`"en" | "de" | "zh" | "es" | "ru"`)
- **`Language`** — used in database models, AI prompts, quiz content (`"中文" | "English" | "Deutsch" | "Español" | "русский"`)

API routes convert between them using the `LANGUAGES` record:

```typescript
const LANGUAGES: Record<string, Language> = { en: "English", de: "Deutsch", zh: "中文", es: "Español" };
```

Note: The `LANGUAGES` record in `api/ai-quiz/route.ts` is currently missing `ru → русский`. This is a known gap.

## next-intl Setup

### Routing (`src/i18n/routing.ts`)

```typescript
export const locales = ["en", "de", "zh", "es", "ru"];
export const routing = defineRouting({
 locales,
 defaultLocale: "en",
});
```

### Middleware (`src/middleware.ts`)

Locale routing is handled by `next-intl/middleware` — it matches all pathnames except `/api`, `/trpc`, `/_next`, `/_vercel`, and file paths with extensions.

### Navigation (`src/i18n/navigation.ts`)

Uses `createNavigation` from `next-intl/navigation` with the routing config. Provides locale-aware `Link`, `useRouter`, `redirect`, etc.

### Request config (`src/i18n/request.ts`)

Provides the `getRequestConfig` for server-side locale resolution.

## Translation Files

Translation JSON files live in `messages/`:

```
messages/
├── en.json
├── de.json
├── zh.json
├── es.json
└── ru.json
```

All 5 files must have the **same key structure**. When adding a new key, add it to all 5 files.

## Using Translations in Components

### Client Components

```typescript
import { useTranslations } from "next-intl";

const t = useTranslations("section-name");
// t("key") returns the translated string
```

### Server Components

```typescript
import { useTranslations } from "next-intl";
// Same API, works in server components within [locale] layout
```

### Getting Current Locale

```typescript
import { useLocale } from "next-intl";
const locale = useLocale(); // Returns "en" | "de" | "zh" | "es" | "ru"
```

## Testing with next-intl

Always mock `next-intl` in tests (see [Testing Patterns](testing.md)):

```typescript
jest.mock("next-intl", () => ({
 useTranslations: () => (key: string) => key,
}));
```

This returns the translation key as the displayed text, so assertions use key strings.
