# Language Learning Methods & Lexiconx Feature Map

> Evidence-based research document mapping proven cognitive science and SLA methods to concrete feature proposals for the Lexiconx vocabulary quiz application.

---

## 1. Retrieval & Retention Methods

These methods target the core mechanism of long-term memory formation: the act of retrieving information strengthens memory traces more than passive review.

| # | Method | Description | Proposed Functionality | Difficulty (1-10) | Feasibility |
|---|--------|-------------|----------------------|------------|-------------|
| 1 | **Active Recall / Retrieval Practice** | The testing effect: retrieving a memory strengthens it more than re-reading (Brown, Roediger & McDaniel, *Make It Stick*; Roediger & Karpicke, 2006). Testing IS learning, not just assessment. | **Free-Recall Mode**: Before showing the multiple-choice quiz, present the target word in a sentence with a blank and ask the user to type the target word from memory (no options). LLM generates the cloze sentence via a new prompt template in `quiz-prompts.ts` keyed as `freeRecall`. The UI renders a text `<input>` instead of option buttons. On submit, compare against `word.word` (case-insensitive, trim whitespace). If correct, proceed to the standard MC quiz for the same sentence; if wrong, show the correct answer with a 3-second delay before continuing. Store result in a new `recallType: 'free' | 'mc'` field on the quiz question response tracked in QuizContext. This doubles the retrieval effort per word — once productive, once receptive. | 5 | **High** — LLM cloze generation is straightforward, text input UI is simple, existing QuizContext handles state. |
| 2 | **Spaced Expansion Retrieval** | Expanding retrieval intervals: test immediately, then at increasing delays (Landauer & Bjork, 1978; *Make It Stick*). The current SM-2 algorithm already handles this at the word level, but NOT within a session. | **Session Micro-Scheduler**: Within a single quiz session, re-present words the user got wrong after a short delay (2-3 questions later), then again after a longer delay (6-8 questions later). Track per-session retest data in a new `sessionRetests: { wordId, dueAfterQuestion, attempts[] }[]` array in QuizContext. The `useQuizManager` hook already manages quiz step progression — extend it to insert retest questions at calculated positions. Words answered correctly on first attempt are NOT retested in-session. This implements Bjork's desirable difficulty of expanded retrieval within a single sitting, complementing the existing across-session SM-2 spacing. | 6 | **High** — Pure client-side logic in QuizContext/useQuizManager, no new API routes, leverages existing quiz step state. |
| 3 | **Interleaving** | Mixing different problem types or categories during practice improves transfer and discrimination vs. blocked practice (Rohrer et al., 2015; *Make It Stick* Ch. 3). Learners must identify WHICH strategy applies, not just apply a known one. | **Category-Mixed Quiz Assembly**: The `/api/words-for-quiz` route currently selects words due for review by `nextReview` date. Modify it to tag each word with its category (`New` | `Learning` | `Mastered` based on `repetitions`/`interval`) and ensure each quiz batch contains a mix: at least 1 word from each available category. Add a `quizComposition: { new: number, learning: number, mastered: number }` field to the quiz generation request. The LLM prompt receives words from different mastery stages, forcing the user to switch between recognizing new vocabulary and discriminating nuanced meanings of familiar words — a qualitatively different cognitive demand. | 4 | **High** — Simple query modification in the API route, existing category logic already in codebase, no model changes. |
| 4 | **Desirable Difficulties (Bjork)** | Making learning harder in the short term improves long-term retention (Bjork, 1994; *Make It Stick*). Includes varying conditions, reducing feedback immediacy, spacing, interleaving. | **Difficulty Toggle with Session Rules**: Add a user preference `difficultyMode: 'standard' | 'challenge'` in the User model's `learningProgress` subdocument. In Challenge mode: (1) remove the `phoneticNotation` hint from quiz sentences for Chinese/Russian, (2) delay correct-answer feedback by 1 question (show a subtle result-pending badge, then reveal at next question), (3) increase distractor plausibility by prompting the LLM to generate near-synonym options. These three toggles are independently controllable. The `quiz-prompts.ts` already accepts level — extend it to accept `difficultyMode` and conditionally suppress phonetic hints and increase distractor difficulty. | 5 | **High** — Mostly prompt modifications, one new field on learningProgress, UI toggle in Settings. |

## 2. Encoding & Elaboration Methods

These methods focus on how information is initially processed and connected to existing knowledge, creating richer memory traces.

| # | Method | Description | Proposed Functionality | Difficulty (1-10) | Feasibility |
|---|--------|-------------|----------------------|------------|-------------|
| 5 | **Elaborative Interrogation** | Asking why questions about a fact deepens encoding by connecting it to prior knowledge (Pressley et al., 1987; *Make It Stick*). Learners generate explanations rather than passively receive them. | **Why This Word Follow-Up**: After the user answers a quiz question correctly, show a collapsible panel: Why does **{word}** mean **{definition}** in this context? The LLM pre-generates a 1-2 sentence etymology or usage explanation as a new `elaboration` field on each `QuizQuestion`. Add `elaboration?: string` to the `QuizQuestion` type. The prompt in `quiz-prompts.ts` adds: For each question, generate a 1-2 sentence elaboration explaining WHY the target word carries its meaning in this context (etymology, word roots, cultural usage, or conceptual link). The user reads this after answering — no action required, but the act of reading an explanation connected to their just-retrieved answer reinforces encoding. | 3 | **High** — One new field on QuizQuestion type, one prompt addition, one collapsible UI element in QuizView. |
| 6 | **Dual Coding (Paivio)** | Information encoded in both verbal and visual form is better remembered (Paivio, 1971). Concrete images paired with words create two retrieval paths. | **AI-Generated Visual Mnemonics**: Add an `imageUrl` field to the Word model (or a `mnemonicImage` field). When a word is first added or generated, call an image generation API (Gemini Imagen via the existing `@google/genai` client) with a prompt: Generate a simple, memorable illustration for the concept of '{word}' ({definition}) — minimalist style, white background, no text. Store the image URL (or base64 in MongoDB). Display the image alongside the word in `WordCard` component and in quiz review panels. For Chinese characters, the image can visually depict the radical components. Cache generated images to avoid regeneration costs. | 7 | **Medium** — Requires image generation API integration, storage strategy (MongoDB base64 or CDN), and new UI in WordCard/QuizView. Gemini Imagen is accessible but adds latency and cost. |
| 7 | **Keyword Method (Atkinson)** | Link a foreign word to a familiar native-language keyword that sounds similar, then create a mental image connecting the keyword to the meaning (Atkinson, 1975; *Fluent Forever* Ch. 4). Proven effective for vocabulary acquisition. | **LLM-Generated Keyword Mnemonics**: Add a `keywordMnemonic: { keyword: string; bridgeSentence: string }` field to the Word model. When a word is created or AI-generated, use the LLM to produce: (1) a native-language keyword that phonetically resembles the target word, and (2) a bridge sentence connecting the keyword to the meaning. Example: Chinese da (big) keyword da bridge A daddy is a big person. Add a new prompt function in `words-prompts.ts` for this. Display in `WordCard` as a collapsible Memory Hook section. In quiz review, show the keyword mnemonic as an optional hint. | 4 | **High** — Simple LLM prompt addition, two new fields on Word model, minor UI additions. The existing `generate-words.ts` pipeline can be extended to include mnemonic generation. |
| 8 | **Generation Effect** | Producing information (rather than reading it) improves retention (Slamecka & Graf, 1978; *Make It Stick*). Self-generated content is remembered better than provided content. | **User-Created Example Sentences**: After mastering a word (interval > 21 days), prompt the user to write their own example sentence using the word. Store in a new `userSentences: { text: string, createdAt: Date }[]` array on the Word model. The LLM evaluates the sentence for grammatical correctness and appropriate usage (1-5 quality score + brief feedback), returning results via a new `/api/evaluate-sentence` route. High-quality sentences get incorporated into future quiz generation as alternative context sentences — the prompt includes them alongside the LLM-generated sentences. Seeing your own sentence in a later quiz is a powerful reinforcer. | 6 | **Medium** — Requires new API route, LLM evaluation prompt, Word model array field, UI for sentence input and feedback display. The evaluation needs guardrails against inappropriate content. |
| 9 | **Chunking** | Grouping individual items into meaningful units reduces cognitive load and expands working memory capacity (Miller, 1956; Cowan, 2001). Vocabulary learned in phrase-level chunks transfers better to production. | **Phrase-Level Vocabulary Units**: Extend the Word model to support a `type: 'word' | 'phrase'` discriminator. The AI word generator (`generate-words.ts` + `words-prompts.ts`) produces collocation chunks alongside single words — e.g., for Chinese, generate bu keqi alongside qing. In quiz generation, the LLM is prompted to use phrase-type entries as indivisible quiz targets. The `WeakestWordsTable` component groups phrases by their component words. Add a filter toggle on the `/cards` page to view words vs. phrases. Phrases have their own independent SRS tracking (separate `interval`/`easeFactor`), so chunk-level and word-level retention are measured independently. | 6 | **Medium** — Requires Word model schema change (type discriminator), prompt updates in both `words-prompts.ts` and `quiz-prompts.ts`, UI filter on cards page. Existing SRS logic applies unchanged to phrases. |

## 3. Input & Comprehension Methods

These methods focus on how learners process and acquire language through exposure, emphasizing the role of comprehensible input and noticing.

| # | Method | Description | Proposed Functionality | Difficulty (1-10) | Feasibility |
|---|--------|-------------|----------------------|------------|-------------|
| 10 | **Comprehensible Input (i+1)** | Acquisition occurs when learners receive input slightly above their current level (Krashen, 1982; Lightbown & Spada, *How Languages are Learned* Ch. 2). The existing level-based prompt system partially addresses this. | **Adaptive Input Complexity Calibration**: The current `quiz-prompts.ts` has 10-level complexity bands (1-10, 11-20, etc.). Refine this into a continuous calibration: after each quiz session, compute an `inputLevelScore` from `correctAnswers / totalQuestions` weighted by word difficulty (using `easeFactor`). If accuracy > 85%, the next quiz uses `level + 2` for sentence complexity (capped at user actual level). If accuracy < 60%, the next quiz uses `level - 3` (floored at 1). Store the offset as `inputLevelOffset: number` on `learningProgress`. This creates a dynamic i+1 zone: the user always encounters sentences at the edge of their comprehension, not stuck at their nominal level. | 5 | **High** — One new field on learningProgress, simple arithmetic in the quiz generation flow, no model changes. |
| 11 | **Input Flooding / Input Enhancement** | Deliberately increasing the frequency or visual prominence of target forms in input text (Sharwood Smith, 1993; *Fluent Forever* Ch. 5). Repeated exposure in varied contexts builds implicit pattern recognition. | **Flooded Reading Passages**: Add a new quiz type: instead of a single sentence per quiz item, generate a short paragraph (4-6 sentences) where 3-5 target words appear 2-3 times each in different grammatical contexts. The LLM prompt explicitly instructs: Write a coherent paragraph where each target word appears at least twice in different syntactic positions. In the UI, target words are bolded and clickable (showing a tooltip with definition + phonetic notation). Questions test comprehension of the paragraph holistically (inference, referent resolution, contextual meaning). This is a new `quizType: 'sentence' | 'passage'` field on the Quiz type. | 6 | **High** — New prompt variant in `quiz-prompts.ts`, new `quizType` field on Quiz type, paragraph rendering in QuizView with tooltip interactivity. All infrastructure exists. |
| 12 | **Noticing Hypothesis (Schmidt)** | Conscious attention to linguistic features is necessary for acquisition (Schmidt, 1990; Lightbown & Spada, Ch. 4). Learners must NOTICE a form before they can acquire it. | **Pattern Highlight Engine**: After the user completes a quiz session, the LLM analyzes the quiz sentences and identifies 2-3 recurring grammatical patterns (e.g., verb aspect markers in Chinese, subjunctive in Spanish). Display these as a Patterns Noticed section in `QuizFinished` — showing the pattern, an example from the quiz, and 1 additional example. Store patterns in a new `noticedPatterns: { pattern: string, examples: string[], firstNoticed: Date, reinforcementCount: number }[]` array on the User model. Over time, the system tracks which patterns the user has been exposed to and ensures future quizzes include sentences featuring un-noticed patterns. This turns incidental exposure into explicit noticing. | 7 | **Medium** — Requires a new LLM prompt for pattern extraction, new array field on User model, UI in QuizFinished, and pattern-tracking logic in quiz word selection. |
| 13 | **Narrow Reading** | Reading multiple texts on the same topic by the same author builds vocabulary through repeated exposure in familiar contexts (Schmitt & Carter, 2000; Krashen, 2004). Reduces cognitive load, increases comprehension. | **Themed Quiz Streaks**: Allow users to select a theme for their quiz session (e.g., Travel, Food, Business, Daily Life). Add a `theme: string` field to the `QuizSession` model and to quiz generation requests. The LLM prompt includes: All sentences must relate to the theme of {theme}. Use vocabulary naturally associated with this domain. When a user completes 3+ sessions on the same theme, award a Themed Reader badge and boost the SRS easeFactor slightly (+0.1) for words learned in themed contexts — they have seen the words in multiple related contexts, strengthening the contextual network. Theme selection UI: a row of pill-shaped buttons above the Generate Quiz button. | 5 | **High** — One new field on QuizSession, prompt parameter addition, simple pill-button UI, minor SRS adjustment. |

## 4. Output & Production Methods

These methods emphasize that producing language (not just comprehending it) drives acquisition by forcing syntactic processing and revealing knowledge gaps.

| # | Method | Description | Proposed Functionality | Difficulty (1-10) | Feasibility |
|---|--------|-------------|----------------------|------------|-------------|
| 14 | **Output Hypothesis (Swain)** | Producing language pushes learners from semantic to syntactic processing — they must construct form, not just interpret it (Swain, 1995; Lightbown & Spada, Ch. 3). Output reveals gaps in knowledge that input alone hides. | **Production Challenge Mode**: A new quiz variant where, instead of selecting an answer, the user hears a sentence via TTS (using the existing `easy-speech` integration) and must TYPE the target word they hear. The LLM generates sentences with the target word removed from the TTS prompt — the user must reconstruct it. For Chinese, this means typing pinyin and selecting the correct character. Add a `productionQuiz` variant to the quiz generation pipeline with a new prompt: Generate a sentence suitable for dictation practice where the target word {word} is the key missing element. Store production accuracy separately in a new `productionAccuracy: number` field on `learningProgress`, enabling the Stats page to show a receptive vs. productive mastery comparison chart. | 8 | **Medium** — Requires new quiz variant type, TTS integration for dictation, input method handling for CJK, typing comparison logic, new stats field. CJK input handling is the hardest part. |
| 15 | **Shadowing** | Immediately repeating heard speech develops pronunciation, intonation, and implicit grammar (Arguelles, 2009; referenced in *Ultralearning* Ch. 4 as drill). Bridges perception and production. | **Shadow and Record Exercise**: After TTS plays a quiz sentence, the user taps a Shadow button and repeats the sentence aloud. Using the Web Speech API SpeechRecognition (available in Chrome/Edge), capture the user speech and compare it to the original sentence using simple string similarity (Levenshtein distance on the target language text). Display a similarity score (0-100%) and highlight mismatched words. Store shadowing attempts in a new `shadowAttempts: { wordId, sentence, similarity, date }[]` array on QuizSession (or a separate model for scale). The UI shows the original sentence, the user recognized text side-by-side, and a Try Again button. Not graded for SRS — this is a voluntary fluency drill. | 7 | **Medium** — Web Speech API has limited browser support (Chrome/Edge only, not Firefox/Safari), requires SpeechRecognition permissions, string comparison logic is simple, but the feature is inaccessible on some browsers. Needs graceful fallback. |
| 16 | **Productive vs. Receptive Recall Asymmetry** | Receptive knowledge (recognizing a word) develops faster than productive knowledge (recalling it to use). This gap persists even at high proficiency (Laufer & Goldstein, 2004). Effective systems must train and measure both independently. | **Dual-Direction Quizzes**: Introduce quiz direction as a parameter: `direction: 'receptive' | 'productive'`. Receptive: see the target language word, select/identify its meaning (current behavior). Productive: see the native-language definition, produce or select the target-language word. Modify `quiz-prompts.ts` to accept a `direction` parameter. In productive mode, the question is phrased in the user native language and the answer options are in the target language (even at low levels — reversing the current level-based language progression). Track receptive vs. productive SRS independently: each Word gets `receptiveSRS: { interval, easeFactor, repetitions }` and `productiveSRS: { interval, easeFactor, repetitions }` (the existing fields become `receptiveSRS`, and new `productiveSRS` starts at interval=0). Quiz sessions alternate direction by default. The Stats page shows two progress curves per language. | 8 | **Medium** — Significant but mechanical refactoring: Word model SRS field split, prompt updates for bidirectional generation, alternating quiz direction logic. The dual SRS tracking is the hardest part but follows the existing SM-2 pattern. |

## 5. Interaction & Feedback Methods

These methods focus on the role of social interaction, corrective feedback, and meta-cognitive awareness in driving acquisition.

| # | Method | Description | Proposed Functionality | Difficulty (1-10) | Feasibility |
|---|--------|-------------|----------------------|------------|-------------|
| 17 | **Interaction Hypothesis (Long)** | Negotiation of meaning during interaction drives acquisition — when communication breaks down, learners are pushed to modify their output (Long, 1996; Lightbown & Spada, Ch. 3). Corrective feedback during negotiation is especially effective. | **Error Explanation and Corrective Feedback Loop**: When a user selects a wrong answer, the quiz does not just show incorrect — it provides a targeted explanation of WHY the distractor is wrong and why the correct answer is right. Add an `errorExplanation: string` field to `QuizQuestion` (pre-generated by LLM). The prompt instructs: For each incorrect option, write one sentence explaining why it is wrong in this context (e.g., it is a false cognate, wrong register, wrong collocation). Display this in a red-bordered panel below the question after a wrong answer. Additionally, the wrong-answer word is flagged for retest in the session micro-scheduler (Method 2). This turns every error into a negotiation-of-meaning episode. | 4 | **High** — One new field on QuizQuestion type, prompt addition, UI panel in QuizView. All infrastructure exists. |
| 18 | **Minimal Pairs Discrimination** | Training learners to distinguish similar-sounding words or phonemes improves both perception and production (Werker & Tees, 1984; *Fluent Forever* Ch. 3). Critical for tonal languages and unfamiliar phonemic contrasts. | **Minimal Pair Quiz Mode**: A specialized quiz type where the LLM generates pairs of words that differ by a single phoneme/tonal feature (e.g., Chinese mai buy vs. mai sell, or Spanish pero vs. perro). The quiz plays both words via TTS and asks Which word did you hear? or presents a sentence with a blank and offers the minimal pair as two of the four options. Add a new `minimalPairPrompt` to `quiz-prompts.ts` that takes a word and generates its minimal pair. The TTS service already exists — call it twice per question. For Chinese, the prompt must be tonal-discrimination aware. Store minimal pair associations in a `minimalPairs: [{ word1Id, word2Id, contrastType: 'tonal' | 'phonemic' | 'orthographic' }]` array — either on the Word model or a separate collection for scalability. | 6 | **Medium** — LLM minimal pair generation requires careful prompting and validation, TTS integration is available, new quiz type needs UI, pair tracking needs storage. Tonal discrimination for Chinese is the key value-add but also the hardest to get right. |
| 19 | **Directness (Ultralearning)** | Learning is most efficient when practiced in the context it will be used (Young, *Ultralearning* Ch. 2). Transfer-appropriate processing: study in the target performance environment. | **Scenario-Based Quiz Modules**: Create topic-based scenarios that simulate real-world language use: ordering at a restaurant, checking into a hotel, making a phone call. Each scenario is a sequence of 3-4 linked quiz items forming a conversation or narrative, where each quiz builds on the previous one context. Add a `scenarioId: string` and `scenarioPosition: number` to the Quiz type. The LLM generates the entire scenario as a coherent dialogue/sequence in a single prompt. The UI presents a conversational flow with a character/avatar and progressive story. Words learned in scenarios get a `contextType: 'scenario' | 'isolated'` tag. The Stats page compares retention rates for scenario-learned vs. isolated words. | 7 | **Medium** — Requires new scenario-aware prompt design, sequential quiz rendering UI, scenario state management. The LLM can generate linked sequences, but ensuring coherence across 3-4 items is non-trivial. Moderate UI complexity. |

## 6. Motivation & Habit Methods

These methods address the affective and behavioral dimensions: without consistent practice, even the best cognitive methods fail.

| # | Method | Description | Proposed Functionality | Difficulty (1-10) | Feasibility |
|---|--------|-------------|----------------------|------------|-------------|
| 20 | **Implementation Intentions** | Specific if-then plans (If X happens, I will Y) dramatically increase goal completion vs. vague intentions (Gollwitzer, 1999; *Atomic Habits* Ch. 3). | **Smart Practice Reminders with If-Then Planning**: When a user streak breaks (currentStreak resets to 0), show a modal: When will you practice tomorrow? with preset options (Morning with coffee, Lunch break, Before bed, Custom time). Store the user if-then plan as `implementationIntent: { trigger: string, time: string, active: boolean }` on `learningProgress`. Use this to schedule browser notification reminders at the chosen time (via the Notification API + service worker). The reminder message incorporates the user own trigger: Time to practice with your morning coffee! When the user completes a session at their planned time, show a Plan Kept! celebration in `QuizFinished`. If the user keeps their plan for 7 consecutive days, boost their streak recovery (allow 1 missed day without breaking streak). | 5 | **High** — New field on learningProgress, browser Notification API integration, modal UI component, simple scheduling logic. Service worker for background notifications adds some complexity. |

## Top 5 Methods by Impact-to-Effort Ratio

Ranking based on: expected retention improvement x user engagement lift / implementation complexity, given the existing Lexiconx stack.

### 1. Elaborative Interrogation (#5) — Impact: High | Effort: Low

**Why 1**: This is the single highest-leverage feature in the entire table. The cost is one `elaboration?: string` field on `QuizQuestion`, one sentence added to the LLM prompt, and one collapsible panel in QuizView. The research evidence is extremely strong — Pressley et al. (1987) found elaborative interrogation produced 2-3x better retention than re-reading, and Dunlosky et al. (2013) rated it as having high utility in their seminal review of learning techniques. It integrates seamlessly into the existing quiz flow without any behavioral change from the user. Every quiz question automatically gets deeper encoding for near-zero marginal cost.

### 2. Error Explanation and Corrective Feedback Loop (#17) — Impact: High | Effort: Low

**Why 2**: Wrong answers are currently a wasted learning opportunity — the user sees incorrect and moves on. The interaction hypothesis (Long, 1996) provides strong evidence that negotiation of meaning during error episodes drives acquisition. The implementation cost mirrors 5: one new field, one prompt addition, one UI panel. The LLM is already generating distractors — explaining why they are wrong is a natural extension. Combined with the session micro-scheduler (2) for retesting wrong words, this creates a complete error-driven learning cycle. The two features together would transform the quiz from an assessment into a learning event.

### 3. Category-Mixed Quiz Assembly (#3) — Impact: High | Effort: Very Low

**Why 3**: Interleaving is one of the most robust findings in cognitive science (Rohrer et al., 2015; *Make It Stick*), yet the current quiz system likely selects words purely by SRS due date, creating de facto blocked practice (all due words are similarly difficult). The fix is a query modification in `/api/words-for-quiz` that enforces category mixing. It requires zero model changes, zero new LLM prompts, and minimal code — yet it forces the user to constantly switch between identify this new word and discriminate this familiar word nuance, which is exactly the cognitive demand that produces robust, transferable memory. The `WeakestWordsTable` already shows category data, so the infrastructure for categorization is complete.

### 4. Keyword Method Mnemonics (#7) — Impact: High | Effort: Low-Medium

**Why 4**: The keyword method is one of the most studied and validated vocabulary acquisition techniques (Atkinson, 1975; Pressley, 1977; *Fluent Forever*). Meta-analyses consistently show medium-to-large effect sizes. For Lexiconx specifically, it addresses a critical gap: the app teaches words in context but has no mechanism for initial word-form-to-meaning anchoring, which is the hardest part of vocabulary learning (especially for non-cognate languages like Chinese). The LLM can generate keyword mnemonics cheaply. Two new fields on Word model, one prompt function in `words-prompts.ts`, one collapsible section in WordCard. The feature is opt-in — users who do not want mnemonics can ignore them — but for learners struggling with new words, it provides an evidence-based scaffold that the current system lacks entirely.

### 5. Adaptive Input Complexity Calibration (#10) — Impact: Medium-High | Effort: Very Low

**Why 5**: The current level system is static — a user at level 35 always gets level-35 sentences. But Krashen i+1 and Bjork desirable difficulties both predict that optimal input is dynamically calibrated: slightly above current ability, adjusting based on recent performance. The implementation is almost trivially simple: compute a post-session accuracy score, store an offset on `learningProgress`, and add/subtract it from the level parameter in quiz generation. No new models, no new prompts, no new UI (it is invisible to the user — they just experience better-calibrated difficulty). The reason it ranks 5 rather than higher is that the existing 10-level complexity bands already approximate i+1 somewhat — this is a refinement, not a new capability. But it is a refinement that costs almost nothing and aligns with one of the most influential theories in SLA.

---

## Implementation Priority Recommendation

**Phase 1 (Quick Wins, 1-2 weeks):** 5 Elaborative Interrogation, 17 Error Explanations, 3 Interleaved Categories, 10 Adaptive Complexity — these four features cost minimal effort, require no architectural changes, and collectively transform the quiz from assessment-only to a genuine learning engine.

**Phase 2 (High-Value Features, 2-4 weeks):** 7 Keyword Mnemonics, 2 Session Micro-Scheduler, 20 Implementation Intentions, 13 Themed Quiz Streaks — these add new capability dimensions (mnemonic anchoring, within-session spacing, habit formation, thematic coherence) with moderate effort.

**Phase 3 (Advanced Features, 4-8 weeks):** 1 Free-Recall Mode, 11 Flooded Reading Passages, 12 Pattern Highlight Engine, 16 Dual-Direction Quizzes — these require deeper architectural changes (new quiz types, model field splits, complex prompts) but address the productive-receptive gap and comprehension gains.

**Phase 4 (Ambitious Features, 8+ weeks):** 6 Visual Mnemonics, 8 User-Created Sentences, 14 Production Challenge Mode, 15 Shadowing, 18 Minimal Pairs, 19 Scenario Modules, 9 Phrase-Level Units — these are high-impact but require significant new infrastructure (image generation, speech recognition, scenario engines, schema migrations).
